# shinji-blog-generator スキル

shinji-ota.com のブログに記事を生成・投稿するスキル。

## Supabase 接続情報

- **Project ID**: `oztfdwbvlarpegozqepn`
- **テーブル**: `public.blog_posts`
- **カラム**: `title` (text), `body` (text), `category` (text), `image_url` (text, nullable), `tags` (text[]), `published` (boolean, default: true)

## カテゴリ一覧

- `労働安全衛生` — 法改正、現場安全、リスクアセスメント、災害事例など
- `AIソリューション` — AI活用事例、最新技術、DX推進など
- `コンサルタント視点` — 実務Tips、考察、業界動向など
- `その他`

## 実行フロー

### Step 1: トピックの決定

ユーザーが特定トピックを指定していれば使う。指定がなければ：
- `WebSearch` で以下を検索する（2〜3件）
  - `"労働安全衛生 最新 2026"` または `"労働災害 対策 最新"`
  - `"AI 安全管理 ソリューション 2026"` または `"職場安全 DX 事例"`
- 検索結果から最も旬なトピックを1つ選び、ユーザーに一言確認する

例：「**『化学物質の自律的管理』に関する記事**を生成しようと思います。よろしいですか？」

### Step 2: 追加リサーチ（必要に応じて）

選んだトピックについて `WebSearch` で詳細情報を1〜2件追加検索し、具体的なデータや事例を収集する。

### Step 3: 画像の準備（オプション）

記事に合わせた画像がある場合、以下のいずれかの方法で対応：

**方法A: GitHub hosting（推奨）**
- ユーザーが画像ファイル（PNG/JPG）をアップロード
- 画像を圧縮（300KB以下推奨）
- `public/images/` フォルダへのコミット方法を提案
- jsDelivr CDN URL形式：
  ```
  https://cdn.jsdelivr.net/gh/dzr01145/shinji_ota@main/public/images/{filename}.jpg
  ```

**方法B: 外部URL**
- ユーザーが既存の画像URL（NotebookLM, Unsplashなど）を提供
- そのURLをそのまま使用

画像がない場合は `image_url` を `null` のまま進める。

### Step 4: 記事生成

以下の形式で記事を生成する。

**執筆スタンス（重要）**：
- 筆者は **太田 真治（Shinji Ota）**。SOMPOリスクマネジメント、技術士（総合技術監理・資源工学）、労働安全コンサルタント
- 鉱山現場13年の経験を持ち、「現場重視」「事例→リスク抽出→PDCA」の視点で書く
- 「事後対応」ではなく「予防」を重視するトーン
- 専門的だが読みやすく、現場の実務者に届く言葉で書く
- **文体は丁寧な日本語（です・ます体）を使用**。「〜です」「〜ます」「〜ではないでしょうか」など、親しみやすく読者に直接語りかける形式。冒頭で読者の課題感に触れるときも「そう思っていらっしゃる方は多いのではないでしょうか」のような敬意を込めた表現を心がける

**記事フォーマット**：
```
タイトル：（30〜50字、具体的・キャッチーに）
カテゴリ：（上記4つから1つ）
タグ：（3〜5個、カンマ区切り）

本文（マークダウン形式）：
## 見出し1
...（本文）...

## 見出し2
...（本文）...

### まとめ
...
```

**本文の品質基準**：
- 600〜1000文字程度（読み応えがありながら読み切れる長さ）
- マークダウン使用：`##` 見出し、`**太字**`、`- リスト`
- 冒頭で読者の課題感に触れる
- 具体的な数字・事例・法令名を含める
- 末尾に「まとめ」または行動を促す一文
- **必ず「## 参考資料」セクションを記事最後に追加する**（以下のStep 4.5を参照）

### Step 4.5: 参考資料セクションの生成

記事本文の最後に「## 参考資料」セクションを追加する。以下のルールに従う：

**参考資料の生成方法**：
1. **Web検索実行**：記事のテーマ・キーワードから、関連する信頼できるリソースを検索する
   - 優先順位：公式サイト（厚生労働省、政府機関）> 専門メディア > 企業・団体サイト
   - 2024年以降の記事を優先
   - 古い情報（1年以上前）は避ける

2. **参考資料の個数**：3～4件を目安に
   - 最低でも3件、最大でも5件を超えない
   - 記事で実際に引用・参照した情報源を優先

3. **フォーマット**：マークダウン形式のハイパーリンク
   ```markdown
   ## 参考資料

   - [出典タイトル（説明あり）](https://duckduckgo.com/?q=出典タイトル)
   ```

4. **タイトル・説明について**：
   - 単なる出典タイトルではなく、**記事テーマに合わせた説明を含める**
   - 例（NG）：`[労働安全衛生法改正まとめ](URL)`
   - 例（OK）：`[2026年4月施行！高年齢労働者への安全対策義務化 | 厚生労働省](URL)`
   - 出典元がわかるよう、メディア名や組織名も含める

5. **リンクの対策**:
   - 元のWebサイトのURLはリンク切れになりやすいため、直接リンクを張るのではなく、記事タイトルで検索できるDuckDuckGoのURL（`https://duckduckgo.com/?q=タイトル`）を使うこと。

**参考資料の配置**：
- 記事の最後（「まとめ」の後）に配置
- 本文の内容と直接関連した資料を選ぶ

```

📝 記事プレビュー
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【タイトル】〇〇〇〇

【カテゴリ】労働安全衛生
【タグ】#タグ1, #タグ2, #タグ3

【本文】
---
（本文全文）
---
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

表示後、以下を伝える：
> 「内容を確認・添削してください。OKであれば **「投稿して」** と言っていただければSupabaseに保存します。修正点があればお伝えください。」

### Step 7: ユーザー確認・添削対応

- ユーザーが修正を求めた場合 → 指摘に従って本文・タイトル・カテゴリ等を修正し、再度プレビューを表示
- ユーザーが「投稿して」「OK」「これでいい」「アップして」等と言った場合 → Step 8へ

### Step 8: サブエージェント（blog-publisher）に投稿を委任

ユーザーの承認を受けたら、**Taskツールを使ってサブエージェントを起動**し、
Supabase投稿とNotebookLM保存の両方を委任する。
親エージェント自身はAPIを呼ばず、サブエージェントの完了を待って結果を受け取る。

**Taskツールのパラメータ：**

- **description**: `"ブログ記事をSupabaseとNotebookLMに投稿"`
- **subagent_type**: `"general-purpose"`
- **prompt**: 以下のテンプレートに記事データを埋め込んで渡す

```
あなたは blog-publisher サブエージェントです。
以下の記事データを受け取り、スキルファイルの指示に従って
SupabaseとNotebookLMに保存してください。

スキルファイル参照先:
/sessions/charming-fervent-mendel/mnt/shinji_ota/.skills/blog-publisher/SKILL.md

=== 記事データ ===
タイトル: （タイトルを記入）
カテゴリ: （カテゴリを記入）
タグ: （タグ1, タグ2, タグ3）
画像URL: （URLまたは「なし」）
投稿日: （今日の日付 YYYY-MM-DD）
本文:
---
（本文全文をマークダウンのまま貼り付け）
---
=================

完了後は PUBLISH_RESULT: 形式で結果を報告してください。
```

**ポイント**: 本文中の特殊文字（シングルクォートなど）はそのまま渡す。
エスケープはサブエージェントのSQL実行時に行う。

### Step 9: 結果の確認と完了通知

サブエージェントから `PUBLISH_RESULT:` 形式の結果を受け取ったら解析し、
ユーザーに報告する。

**成功時:**
> 「✅ 記事を投稿しました！
> - **ブログ**: https://www.shinji-ota.com/blog に反映済み（記事ID: XXX）
> - **NotebookLM**: 『労働安全衛生・AIソリューションに関する最新の考察』にも保存 📓」

**Supabase失敗時:**
> 「❌ Supabaseへの投稿に失敗しました。（エラー内容）
> 再試行しますか？」→ ユーザーに確認してStep 7をやり直す

**NotebookLMのみ失敗時:**
> 「✅ ブログには投稿済みです（記事ID: XXX）
> ⚠️ NotebookLMへの保存は失敗しました。後ほど手動で保存するか、再試行できます。」

## エラーハンドリング

- **Taskツール失敗（サブエージェント起動失敗）**: エラーを表示し、ユーザーに再試行を提案
- **Supabase失敗**: サブエージェントから FAILED 報告 → 再試行するか確認
- **NotebookLMのみ失敗**: ブログ投稿は完了として扱い、ワーニングのみ表示
- **ユーザーが「やめる」「キャンセル」**: Step 6 を実行せずに終了

## 注意事項

- **投稿前に必ずユーザーの明示的な承認を得ること**（Step 7でOKをもらう前にTaskを起動しない）
- 著作権に注意：Web検索結果を直接コピーせず、必ず自分の言葉で再構成する
- 画像は以下の優先度で対応：
  1. ユーザーが提供した画像ファイル（GitHub hostedに変換）
  2. ユーザーが提供した外部URL
  3. 画像なし（`null`）
- サブエージェント（blog-publisher）への指示はPromptに全データを含めること。
  サブエージェントは親のコンテキストを引き継がないため、必要な情報はすべてPromptに書く。

## 親エージェント・サブエージェント設計の意図

```
親 (shinji-blog-generator)         サブ (blog-publisher)
─────────────────────────          ─────────────────────
ユーザーとの対話               →   Taskツールで起動
記事生成・プレビュー           →   Supabase INSERT
承認取得                       →   NotebookLM source_add
結果をユーザーに報告           ←   PUBLISH_RESULT: を返す
```

**なぜ分けるか:**
- 親のコンテキストをAPIコールの詳細で汚さない（ユーザー体験が中心）
- サブエージェントが失敗しても親で再試行ロジックを制御できる
- 将来 Supabase と NotebookLM を2つのサブで並列化できる拡張性がある
